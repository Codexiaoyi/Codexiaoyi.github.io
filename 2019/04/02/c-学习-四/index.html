GC垃圾回收机制

为什么要使用GC呢？也可以说是为什么要使用内存自动管理？有下面的几个原因：

　　1、提高了软件开发的抽象度；

　　2、程序员可以将精力集中在实际的问题上而不用分心来管理内存的问题；

　　3、可以使模块的接口更加的清晰，减小模块间的偶合；

　　4、大大减少了内存人为管理不当所带来的Bug；

　　5、使内存管理更加高效。

　　总的说来就是GC可以使程序员可以从复杂的内存问题中摆脱出来，从而提高了软件开发的速度、质量和安全性。

.net中的GC

.NET中超过80%的资源都是托管资源

非托管资源有：　ApplicationContext, Brush, Component, ComponentDesigner, Container, Context, Cursor, FileStream, Font, Icon, Image, Matrix, Object, OdbcDataReader, OleDBDataReader, Pen, Regex, Socket, StreamWriter, Timer, Tooltip, 文件句柄, GDI资源, 数据库连接等等资源。

简单的三层架构理解

第一层Dal数据层

​ 这一层主要是把与数据有关的函数都封装在这一层，实现数据操作的分离，在其他两层就无需关心与数据库相关操作了。

第二层Bll业务逻辑层

​ 这一层主要是业务逻辑相关的函数封装在这里，做到数据层与UI层的串联以及一些逻辑函数的编写。

第三层UI层

​ 这一层只需要关心与用户交互的一些操作，直接调用Bll层中的业务逻辑函数便可以实现很多业务需求。

Model

​ model层主要是做三层的串联工作，一般来说数据库中的一张表对应model层中的一个类，以此在三层调用的时候，对相应类新建相应的新对象，每一层操作的时候对每个对象进行操作。

分层的目的主要是增强代码的重用性，比如一个写好的项目如果需要更换不同的应用场景，或者说不同的业务需求，则只需要在UI层中做一些界面的修改。并且在写不同需求的时候，与数据库交互的代码也不需要重复写，除开Dal层，在其他层完全实现只针对对象进行操作。既增加了开发效率，又增加了代码的整洁度。

HTTP请求应答模型

“请求/响应”模型是通用的网络模型架构。运用此模型可轻易存取Web资源，简化跨越网络的数据传输操作。它支持HTTP、FTP等通用的网络协议。

HTTP是一种无状态无连接的协议。

1.建立连接。

2.客户端向服务器发出请求。

3.服务器接收到请求，做出相应的响应。

4.断开连接。

多种请求方式

1.GET请求：Get请求返回以URL形式表示的资源，当用户输入一个简单的URL 时，就是使用Get请求。

2.POST请求：Post请求则将表单体置入Web服务器中，发送消息到公告板、新闻组、邮件列表或者其他机构 中，或者为数据处理机制提供诸如提交表单后的结果等数据。

3.HEAD请求：Head请求的 HTTP头所包含的信息与Get请求的响应中的信息相同，可以使用Head请求获取没有发送消息体而由该请求暗指的消息的有关元信息。

筛选器(Filter)

Filter（筛选器）是基于AOP（面向切面编程）的设计，它的作用是对MVC框架处理客户端请求注入额外的逻辑，以非常简单优美的方式实现横切关注点(Cross-cutting Concerns)。横切关注点是指横越应该程序的多个甚至所有模块的功能，经典的横切关注点有日志记录、缓存处理、异常处理和权限验证等。

https://www.cnblogs.com/dlf-myDream/p/5934084.html

其中有关于MVC框架所支持的不同种类的Filter的创建和使用，以及如何控制它们的执行。

模型绑定(Model Binding)

Model Binding允许我们用HTTP请求里的数据创建C#对象，这个对象就是Action

方法的参数。

管道与中间件

当客户端有一个请求POST进入管道，我们需要有一些中间件，比如Logger(日志)中间件=>授权中间件(这里如果失败就会返回相应的错误)=>路由中间件(失败则返回404等错误)，当在管道中经过所有中间件成功后，web服务器会返回一个JSON或者HTML界面，返回的顺序：路由=>授权=>Logger，最后通过网络送到客户端。中间件有很多，MVC框架也是一种，ASP.NET Core会根据中间件的编写顺序而确定其调用顺序。
